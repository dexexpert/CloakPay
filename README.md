# CloakPay

Private payroll and contractor payments for Solana teams, built on the [Cloak](https://cloak.ag) SDK.

## Problem

Every Solana transaction is public by default. For a team paying contractors in USDC, that means salary amounts, recipient relationships, treasury timing, and vendor lists are permanently readable on any block explorer. Public payroll is operationally untenable for crypto-native finance teams, DAOs, agencies, and stablecoin-paying startups.

## Solution

CloakPay lets an operations lead assemble a payroll run, shield treasury funds through Cloak, execute private recipient payouts as a batched private payroll run, and produce a chain-scanned audit trail for approved reviewers through Cloak viewing keys. Privacy is load-bearing — without shielded execution, the product does not solve the problem it sets out to solve.

## Why Cloak Is Central

Cloak is the product mechanism, not a garnish. Three of the SDK's capabilities are wired directly into the user-facing flow:

1. **Shielded treasury deposits.** Before any payout, the treasury runs a `transact(...)` deposit that creates the initial shielded UTXOs (`createUtxo`, `createZeroUtxo`). This decouples the source-of-funds movement from the disbursement itself.
2. **Private payroll transfers.** For each recipient in a batch, `selectUtxos(...)` chooses shielded inputs and `transfer(...)` produces a private transfer to a freshly generated UTXO keypair owned by the claim vault. The recipient's wallet never appears as a destination on the public ledger; the on-chain trace shows opaque shielded activity, not a payroll-shaped outflow.
3. **Viewing-key audit.** Each audit session mints a scoped `vk_*` viewing key. In live mode, the audit report is generated by `scanTransactions(...)` piped through `toComplianceReport(...)` and exported through `formatComplianceCsv(...)` — the report is derived from real chain notes via the viewing key, not from a backend mirror.

Recipient claims close the loop with `fullWithdraw(...)`, moving a private balance to the recipient's destination wallet only when they explicitly claim it.

The integration code lives in:

- [`src/lib/server/runtime.ts`](src/lib/server/runtime.ts) — runtime mode, treasury signer, RPC, and relay resolution.
- [`src/lib/server/cloak-runtime.ts`](src/lib/server/cloak-runtime.ts) — every Cloak SDK call site.
- [`src/lib/server/payroll-service.ts`](src/lib/server/payroll-service.ts) — batch, claim, and audit orchestration.
- [`src/lib/server/state.ts`](src/lib/server/state.ts) — JSON-backed persistence for batches, claims, and stored UTXOs.

## Product Scope

- Operator dashboard for assembling and executing private payroll batches.
- CSV upload (sample at [`src/data/sample-payroll.csv`](src/data/sample-payroll.csv)) and manual recipient entry.
- Shield-fund and private-payout execution states with backend-driven status transitions.
- Recipient-scoped claim page (a contractor only ever sees their own slice of a batch).
- Audit portal with scoped viewing-key sessions, on-demand report generation, and CSV export.
- Persistent backend state and Next.js App Router API routes.
- Live Cloak execution mode with a server-side treasury signer.

## Modes

### Demo mode *(default)*

- Real backend APIs, persistence, claim links, and audit sessions.
- No live Solana or Cloak transactions are submitted; SDK calls are simulated through the same orchestration layer used in live mode.
- Enabled by default for judge-friendly local testing — no funded wallets needed.

### Live mode

- Uses a server-side treasury signer and the Cloak SDK against the configured relay and RPC.
- `SOL`: full live path for shielding, private payroll transfers, recipient withdrawal, and chain-scanned audit.
- `USDC` / `USDT`: live payroll and recipient withdrawal are executed through Cloak relay-backed SPL flows; audit is derived from registered chain notes.

## Local Run

1. Install dependencies:

```bash
npm install
```

2. Create your env file from the template:

```bash
copy .env.example .env.local   # Windows
# or
cp .env.example .env.local      # macOS / Linux
```

Important live-mode variables (leave unset to stay in demo mode):

- `CLOAKPAY_USE_DEMO_MODE=false`
- `CLOAK_TREASURY_KEYPAIR_PATH=...` *or* `CLOAK_TREASURY_SECRET_KEY_JSON=...`
- Optional mint overrides: `CLOAK_USDC_MINT`, `CLOAK_USDT_MINT`
- Optional viewing-key seed: `CLOAK_VIEWING_KEY_NK_HEX`

3. Start the app:

```bash
npm run dev
```

4. Open http://localhost:3000

To reset the local demo to a clean seeded state, delete `.data/cloakpay-state.json` and refresh the dashboard.

## Demo Flow

1. Open `/dashboard`.
2. Upload [`src/data/sample-payroll.csv`](src/data/sample-payroll.csv) (three pre-validated base58 recipient wallets).
3. Review the totals and confidential note.
4. Click **Shield funds and run payroll** — the batch transitions through `shielding` → `completed`.
5. Open a claim link for any recipient and confirm private receipt.
6. Visit `/audit`, generate a scoped viewing key, view the report, and export the audit CSV.

## Stack

- Next.js 15 App Router + React 19
- TypeScript end-to-end
- `@cloak.dev/sdk` for shielded deposits, private transfers, withdrawals, and viewing-key audits
- `@solana/web3.js` for keypair handling and `PublicKey` validation
- Zod for input and CSV validation
- JSON-backed persistence (`.data/cloakpay-state.json`) for hackathon-friendly setup and reproducible demos

## Repo Notes

- Demo mode is on by default so judges can validate the product flow without funded wallets.
- No private keys are stored or committed. `.gitignore` excludes `.env*.local`, the `.data/` directory, and any treasury keypair files.
- Live audit reports are derived from Cloak chain-note scanning in live mode; demo audit reports are deterministic projections of the recorded batch so the UI surface is identical.
- The "batch" run is a batched private payroll execution: one operator action drives a shielded deposit followed by a private `transfer` per recipient. This is intentional — it keeps each recipient's claim independently revealable and revocable through its own viewing scope, instead of binding all recipients to a single transaction.
